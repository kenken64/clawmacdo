# High Security Fixes

This document records how HIGH-01 through HIGH-12 were remediated on the `security-flaw` branch.

## Summary

| ID | Files changed | Fix | Why this is the best fit here | Functional impact |
|----|---------------|-----|-------------------------------|-------------------|
| HIGH-01 | `crates/clawmacdo-ssh/src/ssh.rs`, `crates/clawmacdo-provision/src/provision/commands.rs` | Added stdin-fed SSH execution helpers and switched root/openclaw privilege transitions to `/bin/bash -se` over stdin | This removes the entire fragile nested-quoting class instead of trying to escape ever more carefully | No intended user-visible change; remote command transport is safer and more reliable |
| HIGH-02 | `crates/clawmacdo-core/src/config.rs`, `crates/clawmacdo-cli/src/commands/deploy.rs`, `crates/clawmacdo-cli/src/commands/serve.rs` | Added central hostname normalization and validation and applied it before deploys | One shared validator avoids drift between CLI and web entry points | Invalid hostnames are now rejected early instead of flowing into provisioning |
| HIGH-03 | `crates/clawmacdo-cli/src/commands/deploy.rs` | Removed secret-bearing `--anthropic-api-key`, `--openai-api-key`, and `--gemini-api-key` onboarding arguments | Environment-based secret loading avoids `/proc/<pid>/cmdline` leaks while keeping the existing provisioning flow intact | No normal behavior change if `.env` is present; secrets no longer appear in process listings |
| HIGH-04 | `crates/clawmacdo-cli/src/commands/deploy.rs` | Added local `.tar.gz` inspection, rejected unsafe archive paths, and restored with `tar --no-same-owner --no-same-permissions` into a dedicated temp directory | This blocks traversal and ownership attacks without changing the backup format users already have | Malformed or unsafe archives now fail before restore instead of extracting as root |
| HIGH-05 | `crates/clawmacdo-provision/src/provision/docker.rs`, `crates/clawmacdo-cli/src/commands/deploy.rs` | Removed `openclaw` from the Docker group and force-set sandbox mode to `off` during deploy | Docker group membership is root-equivalent on Linux, so the secure default is to disable sandboxing until there is a brokered non-root runner | If users request sandbox mode during deploy, the deployment now disables it and continues safely |
| HIGH-06 | `crates/clawmacdo-provision/src/provision/user.rs`, `crates/clawmacdo-provision/src/provision/mod.rs` | Replaced wildcard sudoers entries with a minimal exact-command set and updated Tailscale guidance | Least privilege is better than trying to regex-match sudoers shell surfaces | Some manual Tailscale follow-up now explicitly requires a privileged shell instead of passwordless wildcard sudo |
| HIGH-07 | `crates/clawmacdo-core/src/config.rs`, `crates/clawmacdo-cli/src/commands/serve.rs` | Canonicalized backup paths and required them to stay under `~/.clawmacdo/backups` | Canonicalization is the simplest reliable way to block `..`, symlink, and arbitrary path injection in the web API | Web-triggered restore now rejects backups outside the managed backups directory |
| HIGH-08 | `crates/clawmacdo-core/src/config.rs`, `crates/clawmacdo-cli/src/commands/serve.rs` | Canonicalized SSH key paths and required them to stay under `~/.clawmacdo/keys` | This keeps the web API from being used as a file-probe or arbitrary key-loader | Web actions now reject keys outside the managed keys directory |
| HIGH-09 | `crates/clawmacdo-provision/src/provision/openclaw.rs`, `crates/clawmacdo-cli/src/commands/deploy.rs` | Split service runtime secrets into `~/.openclaw/gateway.env` and pointed systemd there instead of `.env` | The service should only inherit what it needs at runtime; setup-only secrets should stay out of the daemon environment | `ANTHROPIC_SETUP_TOKEN` and other setup-only values are no longer inherited by the gateway service |
| HIGH-10 | `crates/clawmacdo-cli/src/commands/deploy.rs`, `crates/clawmacdo-cli/src/commands/serve.rs` | Replaced cloud public-IP unwraps and database mutex unwraps with explicit errors | Reliability fixes should fail closed with useful errors instead of panicking | Missing cloud IPs and poisoned DB locks now return errors rather than crashing handlers or deploys |
| HIGH-11 | `crates/clawmacdo-cli/src/commands/deploy.rs`, `crates/clawmacdo-cli/src/commands/destroy.rs`, `crates/clawmacdo-cli/src/commands/serve.rs` | Removed process-global AWS credential mutation and threaded explicit Lightsail credentials through provider instances | Scoping secrets to child provider calls avoids ambient credential leaks and stale `~/.aws/credentials` behavior | Lightsail actions now rely on explicit request/CLI credentials rather than mutating the process environment |
| HIGH-12 | `crates/clawmacdo-cloud/src/tencent.rs` | Replaced world-open SSH ingress in the Tencent security-group helper with `CLAWMACDO_TENCENT_SSH_CIDR` and a safe default of `127.0.0.1/32` | Safe-by-default is the right baseline for a helper that may later be wired into provisioning | If this helper is used, operators must supply a real SSH CIDR to allow external access |

## Additional Notes

- `cargo check` passes for the full workspace after these changes.
- `cargo test --workspace --no-run` also passes, so test targets compile.
- The main intentional behavior change is sandbox handling: deploys no longer preserve the previous Docker-backed sandbox shortcut because it granted root-equivalent access.
